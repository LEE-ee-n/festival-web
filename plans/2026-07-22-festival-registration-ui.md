## 한눈에 보는 요약

- PC에서는 아티스트 목록을 카드처럼 보이는 행이 아닌 **실제 표**로 바꾼다.
- 표는 `추가 라인업 / 신규 아티스트` → `추가 라인업 / 기존 아티스트` → `기존 라인업 / 기존 아티스트` 순서로 고정한다.
- 모바일에서는 화면이 좁으므로 **카드 형식**을 유지한다.
- 기존 아티스트의 이름·`normalized_name`·별칭은 수정할 수 없고 DB에 저장된 값을 그대로 보여준다.
- 신규 아티스트와 아직 판정 전인 항목만 1단계에서 수정할 수 있다. 2단계부터는 모두 읽기 전용이다.
- 잘못 연결된 기존 아티스트는 정보를 고치는 대신 검색해서 다른 기존 아티스트로 다시 연결한다.
- 먼저 기존 페스티벌 수정 화면에 적용하고 전체 등록을 시험한 뒤 신규 페스티벌 등록 화면에도 적용한다.
- 2~5단계의 전면 디자인 변경, 아티스트 관리 페이지, Bot 수정은 이번 작업에 포함하지 않는다.
- DB 구조는 바꾸지 않으며 자동 테스트·typecheck·PC·모바일 화면·실제 등록으로 확인한다.

완성 모습:

```text
기존 페스티벌 수정 1단계
→ PC 표 / 모바일 카드
→ 기존 아티스트는 DB 정보로 읽기 전용
→ 신규·확인 필요만 수정
→ 전체 명단 확정
→ 2~5단계 진행
→ 실제 기존 축제 수정 테스트
→ 같은 구조를 신규 축제 등록에 적용
→ 실제 신규 축제 등록 테스트
```

## 1. 목적

아티스트가 많아도 한 화면에서 비교할 수 있도록 데스크톱의 등록 1단계를 표 형식으로 바꾸고, 기존 아티스트와 신규 아티스트의 수정 권한을 데이터 상태와 일치시킨다. 기존 DB 아티스트 정보와 OCR 초안 정보가 섞여 보이거나 잘못 저장되지 않게 한다.

## 2. 구현 전 확인한 현재 구조

- 신규 등록과 기존 수정은 `CandidateLineupTab`을 공유한다.
- 기존 수정만 `designVariant="existing-update"`를 사용하며 현재 데스크톱 2열 카드다.
- 같은 컴포넌트의 `mode="timetable"`은 4단계 일정 입력에도 사용되므로, 표 변경은 `mode="review"`에만 적용해야 한다.
- 현재 `matched` 아티스트도 표시 이름·`normalized_name`·별칭 입력칸을 수정할 수 있다.
- 정확 일치와 직접 검색으로 기존 아티스트를 선택해도 DB 별칭 전체를 가져오지 않아 OCR 초안 별칭이 화면에 남을 수 있다.
- 기존 아티스트 직접 선택은 네 번의 `onChange`로 이름·ID·상태를 순서대로 바꿔 중간 상태가 생긴다.
- 기존 수정 2단계의 기존·추가 명단 구분 일부가 `artist_id`가 아니라 `normalized_name` 비교를 사용한다.
- 공통 단계 검증은 `lib/festivals/festivalDraft.ts`, 정확 일치 처리는 `lib/artists/applyNormalizedArtistMatches.ts`가 담당한다.

## 3. 확정할 화면 구조

### 데스크톱 1단계

- 1열 카드 목록을 의미 구조가 있는 HTML 표로 변경한다.
- 각 구역 제목은 표 전체 열을 합쳐 표시한다.
- 열은 `포스터 표기 / 표시 이름 / normalized_name / 별칭 / 작업` 순서로 고정한다.
- 표 구역은 다음 순서로 고정하며 비어 있어도 `없음`을 표시한다.
  1. `추가 라인업 / 신규 아티스트`
  2. `추가 라인업 / 기존 아티스트`
  3. `기존 라인업 / 기존 아티스트`
- 아직 판정되지 않은 항목과 삭제 후보는 세 확정 구역에 섞지 않고 표 상단의 `확인 필요` 구역에 표시한다.
- 제외된 항목은 세 표 구역 아래에 별도로 표시한다.
- 기존 라인업·기존 아티스트 행은 흰 배경과 회색 테두리로 우선순위를 낮춘다.
- 추가 라인업·신규 아티스트·확인 필요 구역은 배경과 테두리로 강조한다.
- 검색 입력과 유사 후보 결과는 선택한 행 바로 아래에서 `표시 이름` 열부터 `별칭` 열까지의 너비로 펼친다. 모바일에서는 전체 너비를 사용한다.
- 최초 표 생성에서 전체 정확 일치 검사를 마치므로 행 내부의 중복 확인 버튼은 두지 않는다. 수정 후 기존 아티스트 확인은 해당 행의 검색·재선택을 사용한다.
- 개별 검색은 `검색창 / 검색 / 신규 아티스트 등록`을 한 줄에 표시한다. 검색 결과가 없을 때만 같은 기존 아티스트가 없다는 문구를 보여준다.
- `신규 아티스트 N명 · 추가 라인업 N명` 요약은 유지한다.
- 제외된 항목은 별도 흐린 행으로 표시하고 `다시 검토`를 제공한다.

구현 방식:

- `CandidateLineupTab.tsx` 안에 1단계 전용 표 행 렌더링을 둔다.
- 하나의 `<table>` 데이터 구조와 이벤트를 사용하고 CSS 반응형 배치만 바꿔 PC와 모바일 상태가 달라지지 않게 한다.
- 데스크톱과 모바일용 입력 컴포넌트를 두 벌 만들지 않는다.

### 모바일 1단계

- 기존처럼 아티스트별 카드 형태로 표시한다.
- 가로 스크롤을 강제하지 않는다.
- 상태, 검색 결과, 편집 권한과 실행 버튼은 데스크톱과 동일한 데이터를 사용한다.

### 2~5단계

- 이번 계획에서는 전면 디자인을 변경하지 않는다.
- 2단계는 1단계 확정 결과를 읽기 전용으로 보여주는 현재 목적을 유지한다.
- 기존 수정 2단계의 기존·추가 판정만 `artist_id` 기준으로 바로잡는다.
- 3~5단계는 1단계 구조 변경으로 인한 회귀가 없는지만 확인한다.

## 4. 아티스트 상태별 권한

| 상태 | 포스터 표기 | 표시 이름 | normalized_name | 별칭 | 검색·재선택 |
|---|---|---|---|---|---|
| 확인 필요 `pending` | 읽기 전용 | 수정 가능 | 수정 가능 | 수정 가능 | 가능 |
| 신규 `new` | 읽기 전용 | 수정 가능 | 수정 가능 | 수정 가능 | 가능 |
| 기존 `matched` | 읽기 전용 | DB 값·읽기 전용 | DB 값·읽기 전용 | DB 값·읽기 전용 | 가능 |
| 제외 `excluded` | 읽기 전용 | 숨김 또는 읽기 전용 | 숨김 또는 읽기 전용 | 숨김 또는 읽기 전용 | `다시 검토` 후 가능 |

세부 규칙:

- `input_name`은 OCR 포스터 원문이므로 직접 수정하지 않는다.
- 직접 추가한 행은 포스터 표기가 없을 수 있으며 표시 이름·`normalized_name`·별칭으로 신규 판정을 진행한다.
- `pending`은 아직 신규·기존 판정 전이므로 OCR 오류 보정을 위해 수정할 수 있다.
- `matched`는 기존 아티스트 마스터 정보를 수정하지 않는다.
- 기존 연결이 틀리면 아티스트 관리 데이터를 고치는 대신 검색에서 다른 기존 아티스트를 선택한다.
- 기존 아티스트 선택 시 `artist_id`, 표시 이름, `normalized_name`, DB 별칭을 한 번에 반영한다.
- 신규 `normalized_name`을 수정하면 `pending`으로 돌아가고 중복 확인을 다시 해야 한다.
- 2단계 이후에는 신규 아티스트도 수정할 수 없다.

## 5. 상태 변경 규칙

```text
OCR·직접 추가
→ pending
   ├─ normalized_name 정확 일치 → matched + artist_id + DB 이름·별칭
   ├─ 검색 결과 직접 선택 → matched + artist_id + DB 이름·별칭
   ├─ 같은 기존 아티스트 없음 선택 → new
   └─ 삭제 선택 → excluded

new에서 normalized_name 수정 → pending → 중복 확인 재실행
matched에서 다른 검색 결과 선택 → 새 matched artist_id와 DB 정보로 원자적 교체
excluded에서 다시 검토 → pending
```

- 기존 수정에서 라인업 상태는 최종 `artist_id`가 현재 축제에 이미 연결됐는지로 `기존 라인업 / 추가 라인업`을 계산한다.
- 신규 아티스트는 `artist_id`가 없으므로 `추가 라인업 / 신규 아티스트`로 표시한다.
- 표시용 이름이나 별칭만으로 기존·추가 라인업을 판정하지 않는다.

## 6. 데이터 처리 변경

- 정확 일치 조회 시 기존 아티스트의 DB 별칭을 함께 가져온다.
- 이름·별칭 검색 결과를 선택할 때도 선택한 아티스트의 DB 별칭을 함께 사용한다.
- `applyNormalizedArtistMatches`가 `matched` 항목의 표시 이름·`normalized_name`·별칭을 DB 값으로 교체한다.
- 기존 아티스트 선택을 여러 번의 일반 필드 변경으로 처리하지 않고, 타입이 정해진 전용 선택 이벤트로 한 번에 반영한다.
- 일반 필드 편집 이벤트는 `pending`과 `new`에서 허용된 필드만 바꾼다.
- 최종 저장에서 기존 아티스트는 계속 `artist_id` 관계만 사용하며 아티스트 마스터 정보를 수정하지 않는다.
- DB 테이블·RPC·migration은 변경하지 않는다.

### 화면 검증

- 데스크톱 표 열이 한눈에 비교 가능하고 페이지 전체 가로 스크롤이 생기지 않는다.
- 모바일 카드가 겹치거나 잘리지 않고 검색 결과가 해당 카드 아래에 표시된다.
- 기존·추가·신규·확인 필요 상태가 색과 문구로 구분된다.
- 기존 아티스트 정보는 읽기 전용이고 DB 별칭이 표시된다.
- 기존 아티스트의 검색·재선택은 가능하며 선택 후 행 전체 정보가 한 번에 바뀐다.
- 신규·확인 필요 항목만 허용된 필드를 수정할 수 있다.
- 신규 `normalized_name` 변경 후 중복 확인 전에는 2단계로 갈 수 없다.
- 제외와 다시 검토가 정상 동작한다.

### 흐름 검증

- 2단계 전체 명단이 1단계 결과와 같고 기존·추가 구분이 `artist_id` 기준으로 정확하다.
- 3단계의 현재값 유지·반영 선택이 변하지 않는다.
- 4단계는 2단계 확정 아티스트만 대상으로 하며 미공개 선택이 유지된다.
- 5단계 전에는 운영 DB가 변경되지 않는다.
- 기존 수정 최종 반영 후 기본정보·추가 라인업·티켓·감사 로그를 확인한다.
- 신규 등록 최종 반영 후 축제·신규 아티스트·라인업·티켓 연결을 확인한다.

실제 최종 등록과 운영 DB 결과 확인에 사용자 로그인 또는 직접 조작이 더 빠르면 정확한 확인 위치와 전달받을 결과를 안내한다.

## 10. 회귀 위험과 대응

| 위험 | 대응 |
|---|---|
| 공유 컴포넌트 변경으로 4단계 타임테이블이 바뀜 | 표 분기는 `mode="review"`에만 적용하고 `mode="timetable"` 렌더링은 유지 |
| PC와 모바일이 서로 다른 상태를 가짐 | 같은 행 데이터·이벤트를 사용하고 CSS 배치만 변경 |
| 기존 아티스트 선택 중 상태가 `pending`으로 되돌아감 | 기존 선택을 전용 이벤트 한 번으로 원자적 반영 |
| OCR 별칭이 기존 아티스트 정보처럼 표시됨 | `matched` 시 DB 이름·별칭으로 교체하고 읽기 전용 표시 |
| 기존 아티스트를 이름으로 비교해 중복 표시 | 기존 라인업·2단계 판정을 `artist_id` 기준으로 통일 |
| 신규 이름 수정 후 중복 확인을 건너뜀 | `normalized_name` 변경 시 `pending` 전환, 공통 단계 검증으로 이동 차단 |
| 표가 좁은 화면에서 깨짐 | 모바일 카드 전환 기준과 긴 문자열 줄바꿈을 localhost에서 확인 |
| 기존 최종 RPC 동작이 달라짐 | DB·RPC는 수정하지 않고 기존 자동 테스트와 실제 1건 최종 반영 확인 |

## 11. 완료 조건

- 작업 단계 1~8의 체크박스가 모두 완료됐다.
- 기존 수정과 신규 등록의 PC 표·모바일 카드가 실제 화면에서 확인됐다.
- 기존 아티스트는 DB 정보로 읽기 전용 표시되고 재선택만 가능하다.
- 신규·확인 필요 아티스트는 1단계에서만 허용 필드를 수정할 수 있다.
- 기존 수정과 신규 등록의 1~5단계 실사용 테스트가 완료됐다.
- 관련 자동 테스트, 변경 파일 ESLint와 typecheck가 통과했다.
- `FESTIVAL_INGESTION_FLOW.md`와 `PROJECT_STATUS.md`가 실제 결과와 일치한다.
- 문서 제목 앞에 `[완료]`를 표시하고 상태와 실제 결과를 기록했다.

## 12. 이번 계획에서 제외

- 2~5단계의 전면 디자인 개편
- 아티스트 관리 페이지 개편
- 티켓 사이트 북마클릿·DB 비교 구현
- Discord Bot 캐러셀 수집 개선

- 계획에 없는 리팩터링과 파일 분리

## 13. 후속 개선점·참고사항

- 2~5단계 디자인은 구체적인 화면 기준을 별도 논의한 뒤 새 계획으로 진행한다.
- 구현 중 이번 계획 밖의 개선점을 발견하면 여기에 기록하고 현재 구현 범위에는 포함하지 않는다.
- 자동 브라우저 환경의 localhost 접근이 정책으로 차단되어 실제 PC·모바일 화면과 최종 등록 검증은 사용자 직접 확인이 필요하다. 우회 검증은 하지 않았다.

## 14. 진행 결과

- 기존 아티스트 정확 일치와 검색 결과에 DB 별칭을 포함했다.
- 기존 아티스트 선택은 ID·표시 이름·`normalized_name`·별칭을 한 번에 반영한다.
- `pending/new`만 1단계 필드를 수정하고 `matched`는 DB 값으로 읽기 전용 표시한다.
- 첫 화면 검토 결과를 반영해 카드형 행을 지정된 세 구역 순서의 실제 PC 표·모바일 카드 구조로 재구성했다.
- 표 열은 `포스터 표기 / 표시 이름 / normalized_name / 별칭 / 작업`으로 고정하고 구역 제목이 5열 전체를 차지하게 했다.
- 미판정·삭제 후보는 `확인 필요`, 제외 항목은 표 아래로 분리했다.
- 기존 수정 2단계의 기존·추가 구분을 `artist_id` 기준으로 변경했다.
- 웹 자동 테스트 112개, typecheck, 변경 파일 ESLint가 통과했다.
- 남은 단계는 localhost 실제 화면 확인과 기존·신규 페스티벌 최종 등록 테스트다.
