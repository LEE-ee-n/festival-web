# 페스티벌 관리 라인업 표 전환 계획

- 작성일: 2026-07-22
- 상태: 코드 구현 완료·실제 화면 확인 대기
- 승인일: 2026-07-22
- 대상: `/admin/festivals/[id]/lineup` 라인업 목록
- 선행 작업: 페스티벌 관리 아티스트 정보 읽기 전용 전환
- DB·저장 로직 변경: 없음

## 한눈에 보는 요약

- 데스크톱의 아티스트별 큰 카드를 한 아티스트당 한 행인 실제 표로 바꾼다.
- 아티스트 읽기 전용 정보는 한 칸에 이름·`normalized_name`·별칭·DB ID를 모아 보여준다.
- 수정 가능한 일정만 입력 요소로 표시한다.
- 모바일에서는 같은 데이터와 이벤트를 카드 형태로 배치한다.

## 현재 문제

- 아티스트마다 연결 상태, 공통정보, 일정 입력이 큰 카드로 반복돼 15명 이상의 명단을 비교하기 어렵다.
- 읽기 전용 정보도 입력창처럼 보여 수정 가능한 일정과 구분이 약하다.
- 공연 날짜·무대 제목이 그룹에 있는데 각 카드가 크게 반복돼 세로 공간을 많이 사용한다.

## 확정 화면 구조

### 데스크톱

- 날짜 → 무대 순서의 현재 그룹 구조를 유지한다.
- 각 무대 그룹 안에 실제 HTML 표를 하나 표시한다.
- 열은 다음 순서로 고정한다.

| 열 | 내용 |
|---|---|
| 아티스트 | 표시 이름, `normalized_name`, 별칭, DB ID·연결 상태 |
| 공연 날짜 | 날짜 입력 |
| 공연 시간 | 시작·종료 시간을 한 칸에 나란히 표시 |
| 무대 | 무대 입력 |
| 상태 | 확정·예정·취소 선택 |
| 작업 | 일정 저장·삭제 |

- 읽기 전용 아티스트 정보에는 입력창 배경과 테두리를 사용하지 않는다.
- 표는 세로 구분선을 최소화하고 가로 구분선 중심으로 표시한다.
- 무대 제목과 인원수는 표 상단 한 줄에 유지한다.
- `기존 아티스트 연결` 큰 박스는 제거하고 DB ID·연결됨을 아티스트 칸의 보조 문구로 축소한다.

### 모바일

- 가로 스크롤 표를 사용하지 않고 한 아티스트씩 카드로 전환한다.
- 같은 행 데이터와 이벤트를 CSS 반응형 배치로 사용하며 데스크톱·모바일 입력 상태를 이중으로 만들지 않는다.
- 아티스트 정보 다음에 날짜·시간·무대·상태·작업을 세로로 표시한다.

## 유지할 동작

- 아티스트 표시 이름·`normalized_name`·별칭 읽기 전용
- 공연 날짜·시작/종료 시간·무대·상태 수정
- 행 단위 `일정 저장`
- 행 단위 `삭제`
- 하단 `미저장 변경사항 전체 저장`
- 발표·정정 기록과 감사 로그
- 기존·신규 라인업 추가 영역

행 저장과 전체 저장이 함께 존재하는 현재 동작은 이번 표 전환에서 바꾸지 않는다.

## 회귀 위험과 대응

- 모바일에서 표가 잘림: 동일 표 행을 좁은 화면에서 블록 카드로 전환한다.
- 날짜·무대 수정 후 그룹 위치가 즉시 바뀜: 현재 `group_date`, `group_stage` 기준 그룹 유지 동작을 그대로 사용한다.
- 시작·종료 입력이 좁음: 시간 열에 최소 너비를 주고 두 입력을 같은 비율로 배치한다.
- 저장 버튼이 다른 행을 저장함: 기존 `row.id` 기반 이벤트를 그대로 전달한다.
- 읽기 전용 전환이 되돌아감: 아티스트 칸에는 입력 이벤트를 추가하지 않는다.
