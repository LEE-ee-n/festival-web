## 한눈에 보는 요약

- 현재 전체 목록은 읽기 전용 표지만 후보 목록과 수정 화면은 큰 카드이므로, PC 화면 전체를 표 중심 구조로 통일한다.
- PC에서는 기존 아티스트 한 명을 한 행에 표시하고, `수정`을 누른 그 행만 입력 가능한 편집 행으로 전환한다.
- 별도로 떨어진 `선택한 아티스트 수정` 카드는 삭제한다.
- 후보 검색 결과도 PC에서는 표로 표시하고, `목록에서 수정`을 누르면 전체 목록의 해당 아티스트 행으로 이동한다.
- 모바일에서는 가로 스크롤 표를 강제하지 않고 후보와 전체 목록을 카드로 표시한다.
- 검색·신규 등록·수정·삭제·정렬·중복 방지와 기존 감사 RPC는 그대로 유지한다.

## 1. 목적

아티스트 수가 늘어나도 표시 이름, `normalized_name`, 별칭을 한눈에 비교하고 원하는 행에서 바로 수정할 수 있게 한다. 페스티벌 등록 화면에서는 기존 아티스트를 읽기 전용으로 유지하고, 아티스트 마스터 정보 수정 책임은 이 페이지에만 둔다.

## 2. 현재 구현과 문제

현재 `/admin/artists`에는 다음 세 영역이 있다.

1. 신규 등록 전 기존 아티스트 검색 폼
2. 검색 후보를 2열 카드로 표시하는 영역
3. 선택한 아티스트를 별도 큰 카드에서 수정하고, 그 아래에 전체 아티스트 읽기 전용 표를 표시하는 영역

문제:

- 검색 후보와 수정 카드가 세로 공간을 크게 사용한다.
- 후보를 선택한 위치와 실제 전체 목록이 분리돼 주변 아티스트와 비교하기 어렵다.
- 전체 목록은 표지만 수정하려면 다시 별도 카드로 이동하므로 표의 장점이 줄어든다.
- 모바일 전용 구조가 없어 현재 최소 너비 820px 표는 작은 화면에서 가로 스크롤이 필요하다.
- 오류·성공 문구가 페이지 공통 영역과 수정 카드 안에 중복 표시된다.

## 3. 확정 범위

### 3.1 상단 검색·신규 등록

- `화면 표시 이름`, `공식 영문명`, `검색`, `초기화` 기능을 유지한다.
- 검색 전에는 신규 등록 버튼을 표시하지 않는다.
- 검색 결과가 0명일 때만 `신규 아티스트 등록` 버튼을 표시한다.
- 신규 등록 성공 후 생성된 아티스트를 전체 목록에서 선택하고 해당 행으로 이동한다.
- 기존 `search_similar_artists`, `create_artist_with_audit` 호출과 `normalized_name` 생성 규칙은 유지한다.

### 3.2 후보 결과

PC 표 열:

| 열 | 내용 |
|---|---|
| ID | 기존 아티스트 ID |
| 표시 이름 | DB의 `artists.name` |
| `normalized_name` | DB 식별값 |
| 유사도 | 검색 RPC 또는 로컬 정확 검색 결과 |
| 작업 | `목록에서 수정` |

- 후보 행 자체에서는 이름·`normalized_name`·별칭을 수정하지 않는다.
- `목록에서 수정`을 누르면 전체 목록 필터 때문에 대상이 숨지 않도록 목록 검색값을 초기화한다.
- 대상 전체 목록 행을 편집 상태로 바꾸고 그 위치로 스크롤한다.
- 후보가 없을 때의 신규 등록 안내만 별도 빈 상태로 표시한다.

### 3.3 전체 아티스트 PC 표

열 순서:

| 열 | 읽기 상태 | 편집 상태 |
|---|---|---|
| ID | 숫자 표시 | 읽기 전용 숫자 |
| 표시 이름 | 텍스트 | 입력칸 |
| `normalized_name` | 고정폭 텍스트 | 영문 소문자·숫자 입력칸 |
| 별칭 | 전체 별칭을 쉼표로 표시 | 쉼표 구분 입력칸 |
| 작업 | `수정` | `저장`·`취소`·`삭제` |

- 한 번에 한 아티스트 행만 편집할 수 있다.
- 별도 `선택한 아티스트 수정` 카드는 제거한다.
- 편집 행은 약한 회색 배경과 진한 가로 테두리로 구분하되 입력칸과 버튼만 강조한다.
- 세로 구분선은 사용하지 않고 가로줄 중심으로 표시한다.
- 기존 기본 정렬 `ID 오름차순`과 ID·표시 이름·`normalized_name` 정렬 버튼을 유지한다.
- 목록 검색은 표시 이름·`normalized_name`·별칭을 계속 검색한다.
- 빈 결과일 때 표 본문 대신 `검색 결과가 없습니다`를 표시한다.

### 3.4 모바일 카드

- `md` 미만에서는 후보와 전체 목록을 한 아티스트당 한 카드로 표시한다.
- 전체 목록 카드에는 ID, 표시 이름, `normalized_name`, 별칭, 작업 버튼을 같은 순서로 표시한다.
- 선택한 카드만 동일한 편집 상태와 `저장`·`취소`·`삭제`를 표시한다.
- PC와 모바일은 같은 조회 데이터, 선택 ID, 편집 값, 저장·삭제 함수를 사용한다.
- 모바일에서 가로 스크롤을 강제하지 않는다.

### 3.5 메시지와 상태

- 오류·성공 메시지는 페이지 상단 공통 위치에 한 번만 표시한다.
- 검색, 신규 등록, 저장, 삭제 중에는 해당 작업 버튼을 비활성화한다.
- 저장 성공 후 행을 읽기 상태로 돌리고 최신 DB 목록을 표시한다.
- `취소`는 DB를 변경하지 않고 편집값을 버린 뒤 읽기 상태로 돌아간다.
- 삭제는 기존 확인창을 유지하며, 라인업에 연결된 아티스트는 기존 RPC가 계속 거부한다.

## 4. 유지할 데이터·권한 규칙

- 기존 아티스트 수정은 아티스트 관리 페이지에서만 가능하다.
- `artists.name`은 사이트 표시 이름이다.
- `artists.normalized_name`은 영문 소문자와 숫자만 허용하며 전체에서 유일해야 한다.
- 별칭은 `artist_aliases`에 한 별칭당 한 행으로 저장하되 화면에서는 쉼표로 입력·표시한다.
- 저장은 `update_artist_admin` 한 트랜잭션과 감사 기록을 계속 사용한다.
- 삭제는 `delete_artist_admin`을 사용하고 연결된 라인업을 자동 삭제하지 않는다.
- 페스티벌 등록·관리 화면의 아티스트 읽기 전용 규칙은 변경하지 않는다.
- 새 `any`·`unknown` 또는 강제 이중 단언을 추가하지 않는다.

## 5. 상태 변화

```text
읽기 행
→ 수정
→ 선택 ID + 현재 이름·normalized_name·별칭을 편집 상태에 복사
   ├─ 저장 → 입력 검증 → update_artist_admin → 목록 재조회 → 읽기 행
   ├─ 취소 → DB 변경 없이 편집 상태 폐기 → 읽기 행
   └─ 삭제 → 확인 → delete_artist_admin → 목록 재조회

후보 결과
→ 목록에서 수정
→ 목록 필터 초기화
→ 해당 전체 목록 행 선택·스크롤

신규 검색 결과 0명
→ 신규 아티스트 등록
→ create_artist_with_audit
→ 목록 재조회
→ 생성된 전체 목록 행 선택·스크롤
```

- 다른 행의 `수정`을 누르면 선택 ID와 편집값을 새 행 기준으로 교체한다.
- 이번 범위에서는 미저장 변경 이탈 확인창을 새로 추가하지 않는다.

## 8. 예외 상황과 처리

- 목록 필터로 선택 대상이 숨겨짐: 후보 또는 신규 등록에서 이동할 때 필터를 초기화한다.
- 저장 후 정렬 위치가 바뀜: DB 재조회와 현재 정렬 규칙을 적용한 뒤 편집 상태를 종료한다.
- 긴 이름·별칭: 셀 안에서 줄바꿈하고 작업 버튼 열은 줄바꿈하지 않는다.
- 중복 `normalized_name`: 현재 클라이언트 선검사와 DB UNIQUE/RPC 오류를 모두 유지한다.
- 한글 이름만 입력해 정규화 값이 비어 있음: 현재 공식 영문명 요구 문구를 유지한다.
- 연결된 아티스트 삭제: DB에서 삭제하지 않고 기존 오류 메시지를 표시한다.
- RPC 저장 실패: 현재 편집값을 유지해 사용자가 수정 후 다시 저장할 수 있게 한다.
- 후보가 전체 목록에 없음: stale 결과로 판단해 오류를 표시하고 목록을 다시 불러온다.

## 9. 회귀 위험

- PC와 모바일에 서로 다른 편집 상태가 생김: 상태와 이벤트는 부모 한 곳에서만 소유한다.
- 입력 중 목록 재조회로 값이 사라짐: 저장·삭제 완료 전에는 자동 재조회를 실행하지 않는다.
- 선택 행 스크롤이 잘못된 행으로 이동함: `artist.id` 기반 ref 또는 안정된 `data-artist-id`를 사용한다.
- 후보 선택이 신규 등록으로 오인됨: 후보에서는 수정 이동만 제공하고 신규 등록은 결과 0명일 때만 제공한다.
- 표 안 버튼이 폼 제출을 일으킴: 모든 행 작업 버튼에 `type="button"`을 명시한다.
- 모바일에서 버튼이나 별칭이 잘림: 카드에서 별칭 줄바꿈과 작업 버튼 wrapping을 확인한다.

## 10. 검증 항목

### 자동 검증

- 변경 단계별 `npm run typecheck`
- 변경 파일 ESLint
- 전체 `npm test`
- 가능한 환경에서 `npm run build`
- 신규 `any`·`unknown`·`as unknown as`가 추가되지 않았는지 검색

### localhost PC

- 후보가 실제 표로 표시됨
- 전체 목록의 한 아티스트가 한 행으로 표시됨
- 선택 행에서만 이름·`normalized_name`·별칭 입력 가능
- 저장·취소·삭제 버튼이 같은 작업 칸에 표시됨
- 검색·ID/이름/`normalized_name` 정렬 유지
- 후보와 신규 등록 결과에서 대상 행으로 정확히 이동
- 긴 이름과 여러 별칭이 옆 행을 침범하지 않음

### localhost 모바일

- 후보와 전체 목록이 카드로 전환됨
- 가로 스크롤 없이 모든 필드 확인 가능
- 선택 카드의 입력과 작업 버튼이 잘리지 않음
- PC와 동일한 검색·수정·취소 상태 사용

### 동작 회귀

- 기존 아티스트 검색 후 수정 행 이동
- 검색 결과가 없을 때만 신규 등록 가능
- 표시 이름 필수 검증
- `normalized_name` 형식·중복 검증
- 별칭 쉼표 입력과 중복 제거
- 수정 저장 후 DB 값 재조회
- 삭제 확인과 연결된 아티스트 삭제 거부

## 11. 완료 조건

- PC에서 후보와 전체 아티스트가 표 중심으로 표시된다.
- 별도 큰 수정 카드 없이 선택한 전체 목록 행에서 바로 수정할 수 있다.
- 모바일에서는 동일 기능을 카드로 사용할 수 있다.
- 검색·신규 등록·수정·삭제·정렬과 감사 기록이 유지된다.
- 자동 검증과 PC·모바일 실제 화면 확인 결과가 문서에 기록된다.

## 12. 제외 범위

- DB 테이블·칼럼·인덱스·RPC·RLS 변경
- 페스티벌 등록·수정 화면 변경
- 아티스트 이미지 업로드
- 페이지네이션·가상 스크롤
- 다중 행 동시 편집 또는 일괄 저장
- 아티스트 병합 기능
- 미저장 변경 이탈 확인창
- 새로운 검색 알고리즘

## 13. 후속 개선점·참고사항

- 아티스트 수가 크게 늘면 서버 페이지네이션과 가상 스크롤을 별도 계획으로 검토한다.
- 중복 후보 두 명을 하나로 합치는 병합 기능은 삭제와 별개인 고위험 작업이므로 별도 계획이 필요하다.
- 운영 중 별칭이 매우 많아지면 별칭 전용 편집 모달 또는 태그 입력 방식을 별도 검토한다.
- Codex 실행 환경에서는 `.env.local` 읽기 권한이 차단돼 production build의 정적 페이지 생성과 localhost 자동 화면 확인을 완료할 수 없다. 컴파일과 TypeScript 단계까지는 통과했다.

## 현재 구현 결과

- PC 후보 결과를 실제 표로 변경했다.
- PC 전체 목록의 선택 행을 이름·`normalized_name`·별칭 편집 행으로 전환했다.
- 별도 `선택한 아티스트 수정` 카드를 제거했다.
- 선택 행에 `저장`·`취소`·`삭제`를 한 작업 칸으로 배치했다.
- 후보 선택과 신규 등록 성공 후 목록 필터를 초기화하고 대상 행으로 이동하게 했다.
- 모바일 후보와 전체 목록을 가로 스크롤 없는 카드로 분리했다.
- `create_artist_with_audit`의 JSON 반환값도 검증 함수로 좁혀 강제 타입 단언을 제거했다.
- 사용자가 localhost에서 PC 표와 선택 행 편집 디자인을 확인했다.
- 사용자가 localhost에서 모바일 카드 표시와 실제 수정 저장 성공을 확인했다.
