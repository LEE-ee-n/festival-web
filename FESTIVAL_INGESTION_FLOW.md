# 페스티봄 축제 수집·등록 기준

이 문서는 Instagram 자료 수집부터 신규 등록 또는 기존 수정의 최종 반영까지 적용하는 영구 업무 규칙과 코드 연결을 기록한다. 진행도와 다음 작업은 `PROJECT_STATUS.md`에서 관리한다.

## 1. 전체 흐름

```text
Instagram URL
→ 게시물 이미지·캡션 추출
→ OCR·JSON 임시자료 생성
→ 축제 식별
   ├─ 완전 일치 없음: 신규 페스티벌 등록
   ├─ 완전 일치 1개: 기존 페스티벌 수정
   └─ 값 누락·오류·복수 일치: 판별 확인 필요
→ 관리자 5단계 검토
→ 마지막 확정에서 운영 DB에 한 번만 반영
```

신규 등록과 기존 수정은 같은 5단계 개념과 공통 검증을 사용하지만 작업 테이블과 최종 RPC는 분리한다.

| 구분 | 작업 테이블 | 작업함 | 최종 RPC |
|---|---|---|---|
| 신규 페스티벌 | `festival_candidates` | `/admin/festival-candidates` | `approve_new_festival_candidate` |
| 기존 페스티벌 수정 | `festival_update_drafts` | `/admin/festival-updates` | `finalize_festival_update_draft` |

## 2. 주요 용어

- `포스터 표기`: 포스터에서 추출한 원문 이름. 신규 연결 시 `festival_artists.input_name`에 최초 저장한다.
- `표시 이름`: 홈페이지에서 보여주는 `artists.name`이다.
- `normalized_name`: 중복 판별과 검색에 사용하는 영문 소문자·숫자 식별값이다.
- `별칭`: `artist_aliases`에 저장하는 다른 표기와 한글 독음이다.
- `기존 라인업`: 해당 축제에 이미 연결된 `artist_id`다.
- `추가 라인업`: DB에는 존재할 수 있지만 해당 축제에는 아직 연결되지 않은 `artist_id` 또는 신규 아티스트다.
- `신규 아티스트`: `artists` DB에 없는 것으로 1단계에서 관리자가 확정한 아티스트다.
- `삭제 후보`: 출연 취소가 자료에 명시된 경우에만 관리자가 판단하는 항목이다. 새 자료에 이름이 없다는 이유만으로 만들지 않는다.
- `임시저장 삭제`: 진행 중인 작업 내용 전체를 삭제한다. 운영 데이터는 변경하지 않는다.

## 3. 축제 식별 기준

아래 세 값이 모두 있고 완전히 같은 승인 축제 한 건만 기존 축제로 판정한다.

```text
festivals.normalized_name
+ festivals.start_date
+ festivals.end_date
```

- 축제 표시명과 `search_aliases`는 검색 보조값이며 자동 판정 기준이 아니다.
- 같은 이름이라도 날짜가 다르면 다른 연도의 신규 축제다.
- 값이 누락되거나 날짜 형식이 잘못되거나 종료일이 시작일보다 빠르면 자동 판정하지 않는다.
- 같은 식별값이 복수로 탐지되면 자동 선택하지 않는다.
- 정확히 일치하지 않는 유사 축제를 자동 병합하지 않는다.

## 4. normalized_name 규칙

### 축제

- `20XX` 연도와 `festival` 단어를 제거한 뒤 영문 소문자와 숫자만 남긴다.
- 예: `ONE UNIVERSE FESTIVAL 2026` → `oneuniverse`

### 아티스트

- 특수문자를 제거하고 영문 소문자와 숫자만 남긴다.
- 자동 연결은 `artists.normalized_name` 100% 일치만 허용한다.
- 표시 이름, 별칭, 유사도 검색은 관리자 선택을 돕는 보조 기능이며 자동 연결 근거가 아니다.
- 실제 축제 관계는 `artist_id`로 저장한다.

축제와 아티스트의 정규화 규칙을 섞지 않는다.

## 5. 공통 5단계

### 1단계: 아티스트 판별·등록

모든 추출 항목을 다음 중 하나로 처리해야 한다.

- 기존 아티스트 연결
- 신규 아티스트 등록
- 아티스트 목록에서 제외

권한과 판정:

- 기존 아티스트는 읽기 전용이다.
- 기존 아티스트의 표시 이름·`normalized_name`·별칭은 OCR 초안이 아니라 DB 값을 보여준다.
- 기존 아티스트 정보 수정은 아티스트 관리 페이지에서만 한다.
- 연결이 잘못됐으면 기존 정보를 고치지 않고 다른 기존 아티스트를 다시 검색·선택한다.
- 신규 아티스트만 표시 이름·`normalized_name`·별칭을 수정할 수 있다.
- 판정 전 `pending` 항목은 OCR 오류 보정을 위해 표시 이름·`normalized_name`·별칭을 수정할 수 있다.
- 포스터 원문 `input_name`은 신규·기존 여부와 관계없이 읽기 전용으로 보존한다.
- `excluded`로 확정한 항목은 변경 미리보기와 최종 RPC 아티스트 배열에서 제외한다.
- 신규 `normalized_name`을 수정하면 중복 확인을 다시 해야 한다.
- 해당 축제에 이미 연결된 `artist_id`는 `기존 라인업 / 기존 아티스트`다.
- DB에는 있지만 해당 축제에 없으면 `추가 라인업 / 기존 아티스트`다.
- DB에도 없으면 `추가 라인업 / 신규 아티스트`다.
- 데스크톱 표는 `추가 라인업 / 신규 아티스트` → `추가 라인업 / 기존 아티스트` → `기존 라인업 / 기존 아티스트` 순서로 표시한다.
- 판정 전 항목은 위 세 구역과 섞지 않고 `확인 필요`에, 제외 항목은 표 아래에 표시한다.
- 확인하지 않은 항목이 있으면 2단계로 갈 수 없다.

### 2단계: 아티스트 최종 확정

- 기존 연결과 추가 명단을 한 화면에서 읽기 전용으로 보여준다.
- 기존 수정의 기존·추가 라인업 판정은 표시 이름이 아니라 `artist_id`를 사용한다.
- 전체 명단을 빠르게 비교하고 확정하는 단계다.
- 아티스트 수정·추가·삭제·재매칭 기능을 두지 않는다.
- 수정이 필요하면 1단계로 돌아간다.
- 1단계를 벗어나면 신규 아티스트 정보도 수정할 수 없다.

### 페스티벌 관리 라인업 권한

- `/admin/festivals/[id]/lineup`은 이미 연결된 라인업의 공연 날짜·시간·무대·상태 관리와 삭제만 담당한다.
- 이 화면에서는 아티스트 검색·신규 등록·라인업 직접 추가를 할 수 없다.
- 아티스트 마스터 생성·수정은 아티스트 관리 페이지에서만 한다.
- 기존 페스티벌의 새 라인업 연결은 이 문서의 5단계 기존 페스티벌 수정 흐름을 사용한다.

### 3단계: 페스티벌 정보·티켓

- 아티스트는 수정할 수 없다.
- 현재 DB 값과 새 값을 비교한다.
- `현재값 유지`는 기존 DB 값을 유지한다.
- `반영 선택됨`은 관리자가 선택한 새 값으로 변경하거나 신규 항목을 추가한다.
- 빈 문자열, 공백, `null`은 기존 값 삭제로 해석하지 않는다.
- 기존 티켓은 유지하고 신규 회차를 추가한다. 충돌 변경은 관리자가 직접 선택한다.

### 4단계: 타임테이블

- 2단계에서 확정한 해당 축제의 아티스트 안에서만 매칭한다.
- 매칭 비교는 `normalized_name`, 저장 관계는 `artist_id`를 사용한다.
- 아티스트 추가·수정·삭제를 할 수 없다.
- 기존 일정은 유지하고 신규 일정 또는 명시적으로 선택한 수정만 반영한다.
- 자료가 없으면 `타임테이블 미공개`를 선택해 전체 미공개로 확정한다.
- 미공개 선택 시 OCR 일정은 저장하지 않는다.
- 선택한 라인업 변경이 있으면 발표·정정 기록의 필수정보를 4단계에서 검사하며, 누락 시 5단계 이동을 차단한다.
- 최종 확정에서도 같은 필수정보를 다시 검사해 이중 안전장치로 유지한다.

### 5단계: 최종 확정

- 앞 단계의 확정 결과를 읽기 전용으로 요약한다.
- `페스티벌 등록 확정` 전에는 운영 데이터를 변경하지 않는다.
- 최종 반영은 한 DB 트랜잭션과 한 감사 이벤트로 처리한다.
- 하나라도 실패하면 운영 변경 전체를 롤백하고 임시작업은 남긴다.
- 성공 결과만 운영 데이터와 감사 로그에 반영한다.

## 6. 신규 등록 규칙

- 신규 등록 작업함은 `work_type = new`만 처리한다.
- 관리자가 최종 승인할 때 축제·신규 아티스트·라인업·티켓을 생성한다.
- 승인 완료 후보는 `status = approved`와 `festival_id`를 이력으로 남긴다.
- 승인 완료 후보는 읽기 전용이며 다시 임시저장 상태로 돌아갈 수 없다.
- 완료 후 OCR·JSON·임시 이미지 메타데이터는 제거하고 출처 URL만 남긴다.

## 7. 기존 수정 규칙

- 기존 기본정보·아티스트 관계·일정·티켓을 먼저 불러온다.
- 기존 자료를 유지하고 관리자가 선택한 추가·변경만 반영한다.
- JSON에 없는 기존 아티스트·티켓·일정을 자동 삭제하지 않는다.
- 같은 아티스트의 다른 공연 날짜는 별도 일정으로 추가할 수 있다.
- 작업 시작 시 기본정보·라인업·타임테이블·티켓의 스냅샷 해시를 저장한다.
- 작업 도중 운영 데이터가 다른 경로에서 바뀌면 최종 반영을 막고 최신 값으로 다시 비교한다.
- 각 단계에서 운영 DB를 따로 저장하지 않고 마지막에 `finalize_festival_update_draft`로 한 번만 반영한다.

## 8. 임시저장·중복·삭제

- 단계와 확정 결과를 서버에 임시저장해 브라우저를 닫아도 이어서 진행한다.
- 사용자가 `임시저장 삭제`를 누르면 그 작업 내용 전체를 삭제한다.
- 같은 URL의 재제출은 허용하지 않는다.
- 신규·기존 작업함 양쪽에서 URL 중복을 확인한다.
- 같은 페스티벌의 기존 작업과 새 작업을 자동 병합하지 않는다.
- 기존 작업이 있으면 삭제 여부를 묻고, 삭제가 확정된 뒤 새 자료로 시작한다.
- Bot과 등록 흐름은 운영 데이터를 자동 삭제하지 않는다.

## 9. OCR·원본·썸네일

- OCR 텍스트와 자동 생성 JSON은 검토 전 임시자료일 뿐 운영 아티스트·타임테이블에 직접 저장하지 않는다.
- 확인할 수 없는 값은 추측하지 않고 빈 값으로 둔다.
- 게시물의 모든 실제 이미지를 추출에 사용한다.
- 관리자가 검토한 첫 이미지만 WebP 정식 썸네일로 승격한다.
- 기존 썸네일 교체는 새 업로드와 DB 반영 성공 후 기존 파일을 정리한다.
- 최종 완료 후 원본 JSON은 남기지 않고 `source_url`만 보존한다.

## 10. Discord Bot

- 원본 위치: `operations/discord-instagram-bot`
- 실행용 외부 Instagram Bot 폴더는 직접 수정하지 않는다.
- 원본에서 테스트한 뒤 `src/bot.js` 등 필요한 파일만 CMD 복사 명령으로 실행 폴더에 반영한다.
- 실행용 Bot의 기존 DPAPI 설정을 유지한다.
- Discord에는 JSON·WebP 파일을 첨부하지 않고 처리 상태와 웹 작업 링크만 보낸다.
- 신규 축제는 신규 등록 작업함, 정확히 일치한 기존 축제는 기존 수정 작업함 링크를 반환한다.

### 페스티벌 관리의 직접 티켓 작업

- `/admin/festivals/[id]/lineup`의 티켓 탭에서는 기존 티켓 RPC로 회차를 추가·수정·삭제한다.
- 직접 티켓 작업 화면에는 별도 변경 출처 URL과 변경 메모 입력을 두지 않는다.
- 감사 이벤트와 변경 전후 데이터는 계속 기록하며 `source_url`, `note`는 비워 둔다.
- 신규·기존 페스티벌 5단계 등록 흐름의 원본 출처 URL 보존 규칙은 그대로 유지한다.

## 12. 필수 회귀 기준

- 같은 `normalized_name`이라도 날짜가 다르면 다른 축제다.
- 세 축제 식별값이 모두 같을 때만 기존 축제다.
- 식별값이 빠지면 신규로 자동 판정하지 않는다.
- 기존 축제가 신규 등록 작업함으로 들어가지 않는다.
- 아티스트 자동 연결은 `normalized_name` 100% 일치만 허용한다.
- 기존 아티스트 정보는 등록 단계에서 수정할 수 없다.
- 신규 아티스트 정보는 1단계에서만 수정할 수 있다.
- 타임테이블은 확정된 축제 아티스트 안에서만 매칭한다.
- `17:50`과 `17:50:00`은 같은 시간이다.
- 빈 신규 값은 기존 값을 지우지 않는다.
- 새 자료에 없다는 이유로 기존 데이터를 삭제하지 않는다.
- 최종 확정 실패 시 운영 DB에 일부 변경이 남지 않는다.
- 기존 수정 결과는 하나의 감사 이벤트에서 확인할 수 있다.

## 13. 문서 갱신 규칙

- 작업 시작 시 `PROJECT_STATUS.md`를 먼저 읽고 이 문서는 축제 등록·수집 작업에만 추가로 읽는다.
- 업무 규칙이나 코드 연결이 바뀌면 이 문서를 먼저 갱신한다.
- 구현과 검증이 끝나면 `PROJECT_STATUS.md`의 진행 상태와 다음 작업을 갱신한다.
- 확정 규칙과 충돌하는 요청은 바로 구현하지 않고 충돌 내용을 사용자에게 알려 판단받는다.
