## 작업 원칙

- Codex 작업 폴더에서 먼저 수정하고 검사한다.
- 필요한 파일만 로컬 테스트 폴더에 복사한다.
- 로컬에서 `.env.local`을 넣어 실제 동작을 확인한 뒤 삭제한다.
- 정상 확인 후 원본 저장소에서 GitHub로 배포한다.

## 1. 기반 환경 및 보안

- Next.js를 16.2.10으로 업그레이드했다.
- ESLint를 Flat Config인 `eslint.config.mjs`로 전환했다.
- 관리자 인증을 `profiles.role = 'admin'`을 확인하는 `is_admin()` 기준으로 통일했다.
- `/admin` 공통 레이아웃에서 관리자 권한을 검사하도록 정리했다.
- 로그인 세션 전체를 출력하던 콘솔 로그를 제거했다.
- 관리자만 축제·아티스트·라인업·티켓·썸네일을 추가·수정·삭제할 수 있도록 RLS를 정리했다.
- 공개 사용자는 승인된 축제와 해당 축제의 공개 라인업·티켓만 읽을 수 있다.

## 2. 아티스트 관리

- 아티스트 검색과 전체 목록을 한 화면에서 제공한다.
- 목록 항목: ID, 화면 표시 이름, `normalized_name`, 별칭, 관리 버튼.
- ID·화면 표시 이름·`normalized_name` 정렬을 지원한다.
- 별칭은 쉼표로 구분해 여러 개를 표시하고 수정할 수 있다.
- 이름·별칭 검색 시 띄어쓰기 차이를 무시한다.
- ID를 제외한 이름·`normalized_name`·별칭을 관리자 RPC로 수정할 수 있다.
- `normalized_name`은 DB에서 필수이며 영문 소문자와 숫자만 허용한다.
- 관련 마이그레이션:
  - `016_require_artist_normalized_name.sql`
  - `017_update_artist_admin.sql`
- 동명이인은 충돌을 피하기 위해 국가 코드를 붙인다.
  - 국내 알리: `alikr`
  - 일본 ALI: `alijp`

## 3. 신규 등록 작업함

- 기존 명칭을 `신규 등록 작업함`으로 변경했다.
- URL은 `/admin/festival-candidates`를 유지한다.
- 직접 신규 작성, 축제 JSON 가져오기, 티켓 수집 JSON 가져오기를 지원한다.
- 검토 대기 초안은 `festival_candidates.draft_json`에 저장한다.
- 기본정보·라인업·티켓을 탭별로 검토한 뒤 승인한다.
- 승인 시 정식 축제와 하위 데이터를 DB 트랜잭션으로 반영한다.

## 4. 축제 JSON 매칭 및 안전 병합

- 자동 매칭 기준은 `normalized_name + 시작일 + 종료일`의 정확한 일치다.
- 정확한 매칭에 실패하면 같은 날짜의 검토 대기 후보를 표시한다.
- 날짜가 같다는 이유만으로 자동 병합하지 않고 관리자가 대상을 선택한다.
- 별도 신규 작업 생성도 관리자가 명시적으로 선택해야 한다.
- JSON을 가져온 뒤 `normalized_name`을 수정할 수 있다.
- 예시:
  - 잘못 추출된 값: `soundplanetatparadisecity`
  - 최종 기준값: `soundplanet`
- 기존 라인업·티켓은 삭제하지 않고 새 항목만 추가하는 안전 병합을 사용한다.
- 비교 상태:
  - 동일
  - 표현 차이
  - 추가
  - 변경 확인
- 변경 확인 항목은 다음 값을 직접 선택한다.
  - 현재 DB 값 유지
  - 가져온 JSON 값으로 변경
- 선택 결과가 화면에 표시되고 실제 저장값에도 반영된다.

## 5. 승인 후 축제 중복 방지

- 검토 대기 후보뿐 아니라 실제 `festivals` 테이블도 중복 검사한다.
- 승인된 축제와 `normalized_name + 시작일 + 종료일`이 같으면 신규 작업 생성을 차단한다.
- `normalized_name`이 달라도 날짜가 같은 정식 축제는 후보로 표시한다.
- 관리자가 `이 정식 축제와 동일`을 선택하거나 기존 축제 관리 화면으로 이동할 수 있다.
- 완전히 다른 축제일 때만 별도 신규 작업을 만들 수 있다.

## 6. 신규 등록 작업함의 아티스트 검토

- 입력값을 React 행 키로 사용해 커서가 사라지던 문제를 수정했다.
- 아티스트와 티켓 입력 중 행이 재생성되지 않는다.
- `normalized_name` 입력 중에는 값을 강제로 정리하지 않는다.
- 입력창을 벗어날 때만 영문 소문자·숫자 형식으로 정리한다.
- 중복 확인은 `normalized_name 전체 중복 확인` 버튼으로 실행한다.
- 상태별 이유를 표시한다.
  - 기존 아티스트 확인: DB의 `normalized_name`과 정확히 일치
  - 신규 아티스트: 정확한 값이 DB에 없음
  - 확인 필요: 값이 비어 있거나 수정 후 재확인 전
- `normalized_name`이 비어 있으면 승인할 수 없다는 경고를 입력칸 아래에 표시한다.
- 확인 필요 또는 신규 판별 항목은 이름으로 유사 후보를 자동 검색한다.
- 유사도 백분율과 기존 아티스트의 `normalized_name`을 표시한다.
- 후보를 클릭하면 기존 아티스트 ID와 연결된다.
- 승인 오류가 발생하면 해당 탭과 문제가 있는 아티스트 입력칸으로 즉시 이동하고 포커스를 준다.

## 7. 기존 축제 승인 이력 보완

- 신규 등록 작업함 도입 전에 생성된 정식 축제는 `festival_candidates` 행이 없어 승인 탭에 나오지 않았다.
- 다음 축제의 기본정보·라인업·티켓 현재 상태를 승인 이력으로 백필했다.
  - Festival ID 2: 2026 매들리 메들리
  - Festival ID 3: 2026 ONE UNIVERSE 페스티벌
  - Festival ID 4: 2026 렛츠락 페스티벌
  - Festival ID 16: 2026 인천 펜타포트 락 페스티벌
  - Festival ID 28: 2026 ASIA TOP ARTIST FESTIVAL
- 생성된 후보 상태는 모두 `approved`다.
- 실제 축제·라인업·티켓 원본은 수정하지 않았다.
- 화면 제목 앞의 `기존 축제 승인 이력 -` 문구는 기능상 필요하지 않아 축제명만 표시하도록 정리했다.
- 관련 마이그레이션: `018_backfill_legacy_festival_candidates.sql`

## 8. 화면 스타일

- 관리자 화면은 흰색 배경을 기본으로 한다.
- 영역은 회색 테두리와 약한 그림자로 구분한다.
- 과도한 하늘색 배경은 사용하지 않는다.
- 관리자 페이지 돌아가기 링크는 연한 회색 버튼으로 표시한다.
- 표의 ID·별칭·관리 열 너비와 구분선을 정리했다.
