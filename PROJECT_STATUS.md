상태 의미:

- `완료`: 구현과 주요 검증이 끝난 상태
- `기능 완료`: 핵심 동작은 확인했지만 추가 회귀 시험이 필요한 상태
- `진행 중`: 구현됐으나 실제 전체 흐름 검증이 끝나지 않은 상태
- `아이디어`: 방향만 정했고 아직 구현하지 않은 상태

## 3. 페스티벌 등록의 현재 구조

신규 등록과 기존 수정은 공통 5단계를 사용하지만 작업함과 최종 저장 RPC를 분리한다.

1. 아티스트 판별·등록
2. 아티스트 최종 확정
3. 페스티벌 정보·티켓
4. 타임테이블 또는 전체 미공개
5. 최종 검토·등록 확정

핵심 안전장치:

- OCR 결과를 아티스트·라인업·타임테이블에 바로 저장하지 않는다.
- 아티스트 자동 연결은 `normalized_name` 100% 일치만 허용한다.
- 실제 관계는 `artist_id`로 저장한다.
- 타임테이블은 확정된 해당 축제 아티스트 안에서만 매칭한다.
- 빈 신규 값으로 기존 값을 지우지 않는다.
- 자동 삭제와 자동 병합을 하지 않는다.
- 마지막 확정 전에는 운영 데이터를 변경하지 않는다.
- 최종 반영은 한 트랜잭션과 한 감사 이벤트로 처리한다.

세부 규칙과 코드 연결: [FESTIVAL_INGESTION_FLOW.md](FESTIVAL_INGESTION_FLOW.md)

## 4. 아티스트 수정 권한

- 신규 아티스트만 1단계에서 표시 이름·`normalized_name`·별칭을 수정할 수 있다.
- 신규 아티스트도 2단계부터는 수정할 수 없다.
- 기존 아티스트는 페스티벌 등록 전체 단계에서 읽기 전용이다.
- 기존 아티스트 정보 수정은 아티스트 관리 페이지에서만 한다.
- 기존 아티스트 연결이 잘못된 경우 1단계에서 다른 기존 아티스트를 다시 선택한다.

이 권한 구분은 신규·기존 페스티벌 등록에 동일하게 적용한다.

## 5. 최근 주요 업데이트

### 단계형 등록과 DB

- 신규 등록과 기존 수정을 서로 다른 작업함으로 분리했다.
- 신규 등록은 `festival_candidates`와 `approve_new_festival_candidate`를 사용한다.
- 기존 수정은 `festival_update_drafts`와 `finalize_festival_update_draft`를 사용한다.
- 임시저장, 단계 확정 상태, 선택 결과와 스냅샷 해시를 저장한다.
- 같은 축제의 기존 작업과 새 작업을 자동 병합하지 않는다.
- 승인 완료된 신규 후보는 읽기 전용이며 다시 검토 대기로 돌아가지 않는다.

### Discord Bot

- Instagram URL을 분석한 뒤 신규 축제와 기존 축제를 구분한다.
- 신규 축제는 신규 등록 작업함으로 보낸다.
- 축제 `normalized_name`과 날짜가 정확히 일치하면 기존 수정 작업함으로 보낸다.
- JSON·WebP 파일을 Discord에 첨부하지 않고 작업 링크와 상태만 반환한다.
- 같은 URL의 중복 처리를 신규·기존 작업함 양쪽에서 확인한다.
- Bot 원본은 `operations/discord-instagram-bot`이며 실행 폴더는 검증 후 CMD로 복사한다.

### 관리자 UI

- 등록 1단계 데스크톱 목록을 실제 표로 변경했다.
- 표는 `추가 라인업 / 신규 아티스트`, `추가 라인업 / 기존 아티스트`, `기존 라인업 / 기존 아티스트` 순서로 구역을 고정했다.
- 확인이 끝나지 않은 항목은 표 상단의 `확인 필요`, 제외 항목은 표 아래에 분리했다.
- 기존 항목은 시각적 강조를 줄이고 추가 라인업·신규 아티스트를 강조했다.
- `신규 아티스트 N명 · 추가 라인업 N명` 요약을 추가했다.
- 아티스트 검색·재선택 뒤 상태가 잘못 초기화되던 오류를 수정했다.
- 승인 완료 화면에서 임시저장과 수정 기능을 제거했다.
- 등록 1단계는 PC 표·모바일 카드 구조를 공유한다.
- 기존 아티스트는 DB 이름·`normalized_name`·별칭을 읽기 전용으로 표시하고 재선택만 허용한다.
- `pending/new` 항목만 1단계에서 수정하며 포스터 원문은 읽기 전용으로 유지한다.
- 페스티벌 관리의 기존 아티스트 표시 이름·`normalized_name`·별칭을 읽기 전용으로 변경하고 일정 저장과 아티스트 마스터 수정을 분리했다.
- 페스티벌 관리 라인업을 데스크톱 표·모바일 카드 구조로 변경했다.
- 페스티벌 관리의 아티스트 검색·신규 등록·라인업 직접 추가 UI와 관련 상태를 제거했다. 새 라인업 연결은 5단계 기존 페스티벌 수정 흐름에서만 처리한다.
- 페스티벌 관리 티켓 탭에서 실제 사용 기록이 0건이었던 변경 출처 URL·변경 메모 입력과 관련 상태를 제거했다. 티켓 감사 이벤트와 변경 전후 기록은 유지한다.

## 6. 현재 디자인 적용 상태

- PC는 실제 표, 모바일은 같은 데이터의 카드 형식을 사용한다.
- 표의 기본 열은 `포스터 표기 / 표시 이름 / normalized_name / 별칭 / 작업`이다.
- 기존 라인업·기존 아티스트는 눈에 덜 띄게 표시한다.
- 추가 라인업·신규 아티스트·확인 필요 행은 강조한다.
- 유사 아티스트 검색 결과는 해당 행 아래에서 확인하도록 구성한다.
- 수정 권한은 화면 형식과 관계없이 4절의 규칙을 유지한다.

## 8. 완료한 검증

- 신규 페스티벌 등록을 실제 화면에서 최종 승인까지 완료했다.
- 승인 완료 후보가 다시 검토 대기로 돌아가던 문제를 수정하고 재확인했다.
- 기존 축제 자료가 기존 수정 작업함으로 분리되는 것을 확인했다.
- 기존 수정 1단계의 아티스트 검색·재선택 오류를 수정했다.
- 기존 페스티벌 수정 최종 반영을 완료했다: 감사 작업 `#350`, 변경 `18건`.
- 삭제·제외한 빈 아티스트가 최종 RPC에 포함되던 오류를 수정하고 실제 반영 성공을 확인했다.
- 웹 타입 검사 통과
- 변경 파일 ESLint 통과
- 웹 자동 테스트 112개 통과
- Discord Bot 자동 테스트 9개 통과

프로젝트 전체 린트는 과거 파일과 자동 생성 파일의 별도 문제가 있어 전체 무오류로 기록하지 않는다.

## 9. 아직 완료하지 않은 검증

1. 최종 반영 후 기본정보·추가 라인업·티켓·감사 로그 `#350` 상세 확인
3. 기존 썸네일 교체 후 기존·임시 Storage 파일 정리 확인
4. Discord Bot의 실제 게시물 이미지 수에 맞춘 캐러셀 수집 확인
5. Bot의 기존 축제 조회에서 승인 상태 조건 검토
6. 공개 화면의 실제 PC·모바일 배포 회귀 테스트
7. 기존 업데이트를 우회하는 과거 직접 JSON 경로의 차단 여부 검토
8. 전체 구현에 대한 `code-review-excellence` 기준 읽기 전용 검토

## 10. 오늘 이후 작업 순서

1. 실제 화면에서 PC 표의 세 구역 순서와 모바일 카드 전환 확인
2. 기존 수정 2~5단계 회귀와 전체 흐름을 실제 페스티벌로 최종 시험
3. 다른 신규 페스티벌을 실제 추가하며 전체 흐름 재시험
4. 2~5단계 디자인과 아티스트 관리 페이지 개편은 각각 별도 계획으로 진행

현재 코드 적용과 자동 검증, 기존 페스티벌 수정 최종 반영 1건은 완료했다. 자동 브라우저의 localhost 접근 제한으로 실제 모바일 화면과 다른 신규 페스티벌 등록은 사용자 확인이 남아 있다.

등록을 막는 오류를 먼저 수정하고, 자잘한 디자인 문제는 별도 개선 목록으로 남긴다.

## 11. 이후 아이디어

### 티켓 사이트 누락 확인

- YES24·NOL·Ticketlink의 전체 목록을 북마클릿으로 수집한다.
- URL과 축제 식별값을 기존 DB와 비교한다.
- 결과를 `등록됨 / 미등록 / 확인 필요`로 나눈다.
- 미등록 항목의 Instagram 원문은 관리자가 확인한 뒤 현재 5단계 등록 흐름에 넣는다.
- 티켓 사이트 상세정보의 자동 등록은 당장 구현하지 않는다.

### URL 처리 이력

나중에 원본 URL별 처리 상태를 다음과 같이 관리하는 방안을 검토한다.

- 미변환
- JSON 생성됨
- 등록 작업 중
- 등록 완료
- 제외

현재는 아이디어만 확정했으며 구현하지 않았다.

## 12. 공개 전 우선 확인

- 기존 페스티벌 수정 1건을 최종 반영까지 성공시킨다.
- 과거 직접 업데이트 경로가 5단계 안전장치를 우회하지 않는지 확인한다.
- 실패 시 운영 DB가 일부만 변경되지 않는지 확인한다.
- Supabase 백업·복구 방법을 확인한다.
- 공개 화면을 PC와 모바일에서 점검한다.
- Bot 중단·로그인 만료를 알아볼 최소한의 상태 확인 방법을 마련한다.
