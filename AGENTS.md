# Festibom 작업 규칙

## 기본 원칙

- 가능한 답변은 간결하게 작성한다.
- 사용자가 설명만 요청하면 코드를 수정하지 않는다.
- 요청 범위 밖의 기능을 임의로 추가하지 않는다. 발견한 문제는 문서의 정리 대상으로 기록한다.
- 사용자 변경사항을 보존하고 관련 없는 파일은 수정하지 않는다.

## 단일 원본

- 홈페이지와 운영 코드의 원본은 `C:\Users\소닉스\Documents\festibom`이다.
- Discord Bot 원본은 `operations/discord-instagram-bot`이다.
- 외부 Instagram Bot 폴더는 실행용 복사본이며 직접 수정하지 않는다.
- Bot 변경은 원본에서 검증한 뒤 ZIP이 아니라 CMD 복사 명령으로 실행용 폴더에 반영한다.
- 프로젝트 루트의 `.env.local`은 홈페이지 실행에만 사용한다. 내용을 읽거나 출력하거나 문서에 기록하지 않는다.
- Discord Bot 인증은 기존 DPAPI 설정을 유지하며 다시 구성하거나 `.env.local`과 합치지 않는다.

## 작업 시작 순서

1. 요청을 설명·진단·구현·DB 변경 중 하나로 분류한다.
2. `PROJECT_STATUS.md`에서 현재 상태와 다음 작업을 먼저 확인한다.
3. 요청에 해당하는 참고문서만 추가로 읽는다.
4. 관련 공통 함수, 타입, 화면, 테스트, migration을 범위 검색한다.
5. 기존 구현으로 해결 가능한지 먼저 판단한다.
6. 수정 범위와 완료 조건을 정한 뒤 작업한다.

관련 문서:

- 전체 진행도·업데이트·다음 작업: `PROJECT_STATUS.md`
- 축제 수집·등록 규칙과 코드 연결: `FESTIVAL_INGESTION_FLOW.md`
- 운영 DB 칼럼·RPC·감사 로그: `DATABASE.md`
- 아티스트 관리: `ARTIST_MANAGEMENT.md`
- 프로젝트 파일 구조: `ARCHITECTURE.md`
- 인증·RLS·Storage 보안: `SECURITY.md`
- 티켓 목록 크롤러: `CRAWLER_REQUIREMENTS.md`

## 구현 순서

- 업무 흐름 변경은 `흐름 문서 → 공통 규칙 → 테스트 → 화면·Bot 연결 → 전체 검증` 순서로 진행한다.
- 사용자가 승인한 계획 범위는 일부만 구현하거나 임의로 다음 작업으로 미루지 않고 전부 구현한다.
- 승인된 계획의 모든 작업과 단계가 완료될 때까지 계속 진행한다. 단, 사용자가 중지를 요청하거나 권한·로그인·외부 상태로 실제 진행이 불가능한 경우에는 중지 사유와 남은 단계를 보고한다.
- 같은 목적의 함수나 타입이 있으면 새로 만들지 말고 기존 공통 구현을 확장한다.
- 공통 규칙을 화면이나 Bot 내부에 다시 구현하지 않는다.
- 구현 코드에 `any` 또는 `unknown` 타입을 새로 사용하지 않는다. 외부 입력은 구체적인 타입과 검증 함수로 좁힌다.
- 신규 축제 등록과 기존 축제 업데이트 흐름을 섞지 않는다.
- 자동 삭제를 구현하지 않는다. 삭제 후보 표시와 실제 삭제 결정을 분리한다.
- 축제 `normalized_name`은 `20XX` 연도와 `festival` 단어를 제거한 뒤 영문 소문자와 숫자만 남긴다.
- 아티스트 `normalized_name`은 특수문자를 제거하고 영문 소문자와 숫자만 남긴다. 축제 규칙과 섞지 않는다.
- 포스터의 아티스트 원문 표기는 `input_name`에 보존한다.
- 변경 범위와 무관한 프로젝트 전체 조사를 매번 반복하지 않는다.

## 계획 문서 작성 규칙

- 새 로직·함수·컴포넌트를 만들거나 페이지 구조, 데이터 흐름, 권한, 동작을 변경하는 작업은 구현 전에 계획 MD를 작성한다.
- 카드형을 표형으로 바꾸는 것처럼 화면 구조를 변경하는 작업도 계획 대상이다.
- 색상, 글자 크기, 여백, 간격, 테두리처럼 동작과 구조를 바꾸지 않는 순수 시각 조정은 계획 문서 없이 진행할 수 있다.
- 계획 작성 전에 관련 현재 코드·공통 함수·타입·테스트·문서와 직접 영향 범위를 충분히 조사한다.
- 계획에는 예상 수정 파일과 컴포넌트, 상태 변화, 예외 상황, 기존 규칙과의 충돌, 회귀 위험, 검증 방법을 구체적으로 포함한다.
- 구현 중 추가 판단이 생기지 않도록 불명확한 요구와 기술적 불확실성을 계획 검토 요청 전에 최대한 해소한다.
- 모호한 작업명이나 나중에 정한다는 항목을 남기지 않고 실제 구현 가능한 단위로 계획을 작성한다.
- 계획 대상 구현은 사용자가 승인한 계획 문서에 적힌 범위로만 진행한다.
- 구현 중 계획에 없는 개선점이나 참고사항을 발견해도 재승인을 요청하거나 범위를 넓히지 않고 승인된 계획을 먼저 끝낸다.
- 계획 밖에서 발견한 내용은 계획 문서의 `후속 개선점·참고사항`에 기록하고 이후 별도 계획 대상으로 남긴다.
- 계획 문서는 `plans/YYYY-MM-DD-작업명.md` 형식으로 작성해 작업 날짜를 알 수 있게 한다.
- 계획에는 목적, 확정 범위, 작업 순서, 유지할 규칙, 검증 항목, 완료 조건, 제외 범위, 후속 개선점·참고사항을 기록한다.
- 계획 문서를 작성한 뒤 사용자에게 검토를 요청하고 명시적인 승인 전에는 계획 대상 구현을 시작하지 않는다.
- 사용자가 계획 문서에 주석이나 수정 의견을 남기면 먼저 계획을 수정하고, 수정된 계획을 기준으로 구현한다.
- 진행 중인 계획은 `PROJECT_STATUS.md`에서 연결한다.
- 계획의 작업이나 단계가 완료될 때마다 해당 계획 문서에 즉시 완료 표시를 남긴다.
- 작업 완료 후 계획 문서 제목 앞에 `[완료]`를 표시하고 상태와 실제 결과를 갱신한다.
- 완료된 계획의 검증 결과와 다음 작업을 `PROJECT_STATUS.md`에 반영한다.
- 새 요청이 진행 중인 계획이나 확정 규칙과 충돌하면 구현 전에 충돌 내용을 사용자에게 알리고 판단받는다.

## 흐름 문서 작성 규칙

- 새 페이지, 함수, API, RPC, 테이블 또는 주요 기능을 만들거나 기존 처리 흐름을 변경하면 관련 흐름 문서를 반드시 작성하거나 갱신한다.
- 문서에는 입력부터 최종 저장까지의 단계, 사용 페이지, 공통 함수, DB 테이블·칼럼·RPC, 예외 분기와 관리자 검수 지점을 기록한다.
- `현재 구현`, `목표 흐름`, `남은 불일치`를 구분한다.
- 구현을 마치기 전 문서와 실제 코드 연결이 일치하는지 확인한다.
- 작업이 끝나면 관련 참고문서를 먼저 갱신하고 `PROJECT_STATUS.md`에 진행 상태·검증 결과·다음 작업을 요약한다.
- 새 MD 파일을 만들기 전에 기존 참고문서에 합칠 수 있는지 먼저 확인한다.

## DB 변경 규칙

- DB를 변경하지 않는 작업에서는 DB 전체 구조를 다시 검사하지 않는다.
- 테이블·칼럼·제약조건·인덱스·RPC를 추가하거나 수정할 때만 `DATABASE.md`, 관련 migration, 관련 코드 사용처를 범위 검색한다.
- 새 DB 객체를 만들기 전에 같은 목적이나 유사한 이름의 기존 객체가 없는지 확인한다.
- 기존 객체로 해결할 수 있으면 새로운 테이블·칼럼·RPC를 만들지 않는다.
- migration 번호 중복, 동일 객체 중복 생성, 기존 데이터 영향, RLS와 권한을 확인한다.
- migration 파일 작성만으로 운영 DB에 적용됐다고 간주하지 않는다.
- 원격 DB 전체 점검은 migration 적용, 스키마 불일치 진단, 배포 전 검증이 필요할 때만 수행한다.
- DB 변경 후 관련 테스트를 실행하고 `DATABASE.md`와 흐름 문서를 실제 상태로 갱신한다.
- 한글이 포함된 SQL·JSON·문서를 클립보드로 복사할 때 `cmd type | clip`을 사용하지 않는다. 반드시 PowerShell의 `Get-Content -Raw -Encoding UTF8 | Set-Clipboard`를 사용한다.
- SQL을 사용자에게 전달하기 전에 UTF-8로 다시 읽어 한글 문자열과 따옴표가 깨지지 않았는지 확인한다.

## 검증 규칙

- 먼저 변경한 공통 함수의 단위 테스트를 실행한다.
- 다음으로 관련 기능 테스트를 실행한다.
- 의미 있는 구현 단계가 끝날 때마다 typecheck를 실행해 새 타입 오류를 즉시 수정한다.
- 영향 범위가 넓으면 전체 테스트와 타입 검사를 실행한다.
- 화면 변경은 localhost에서 실제 화면과 오류를 확인한다.
- DB 변경은 적용 전 SQL과 영향 범위를 검토하고, 적용 후 실제 스키마 또는 동작을 확인한다.
- Chrome 프로필처럼 자동 생성된 외부 파일의 린트 오류를 작성 코드 오류와 구분한다.
- 검증하지 못한 항목은 완료라고 표현하지 않고 이유를 명시한다.

## 사용자 요청이 더 빠른 경우

- 추가 권한, 로그인, 비밀정보 또는 사용자 계정의 직접 조작이 필요하면 우회하거나 추측하지 말고 즉시 사용자에게 요청한다.
- 도구로 오래 우회하는 것보다 사용자가 직접 실행·확인하는 편이 빠르고 정확하면 사용자 행동을 요청한다.
- 요청할 때는 이유, 실행 위치, 명령 또는 확인할 값, 전달받아야 할 결과를 명확히 적는다.
- 같은 권한 문제나 실패를 반복해서 시도하지 않는다.

## 작업 완료 조건

- 요청한 기능이 실제 코드에 반영됐다.
- 관련 테스트와 필요한 타입·화면·DB 검증이 끝났다.
- 흐름 문서와 DB 문서가 실제 구현과 일치한다.
- 신규·기존·미사용 또는 정리 후보가 명확히 구분됐다.
- 완료 보고에는 수정 파일, 검증 결과, 남은 문제만 간결하게 적는다.

## 사용량 및 반복 작업 절약

- 설명은 사용자가 판단하거나 실행하는 데 필요한 내용만 간결하게 작성한다.
- 전체 로그나 대용량 파일을 그대로 출력하지 않고 필요한 줄만 검색한다.
- 이미 확인하여 문서에 기록한 내용은 변경 근거가 없으면 다시 검사하지 않는다.
- 전체 검사보다 변경 파일과 직접 영향 범위를 먼저 확인한다.
- 사용자에게 실행 명령을 줄 때는 현재 위치에 의존하지 않는 절대경로를 사용한다.
- 실제 URL 수집처럼 비용이 큰 검증은 로컬 단위 테스트가 모두 통과한 뒤 마지막에 한 번만 수행한다.
- 실패한 명령을 그대로 반복하지 않고 원인을 수정한 후 재시도한다.
