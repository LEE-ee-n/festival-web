# 신규 등록 작업함

## 목적

신규 페스티벌 등록과 수집자료 검토를 한 경로에서 처리한다. 정식 등록 전 자료는 `festival_candidates`에 작업 초안으로 보관하고, 승인할 때만 실제 축제·라인업·티켓에 반영한다.

## 진입 경로

- 관리자 첫 화면의 `신규 등록 작업함`
- 실제 URL: `/admin/festival-candidates`

기존 `/admin/festivals/new`는 관리자 첫 화면에서 제거했다. 새 작업은 신규 등록 작업함의 `직접 신규 작성`으로 시작한다.

## 작업 시작 방법

- 직접 신규 작성
- 사진·게시글 기반 축제 JSON 가져오기
- 티켓 사이트 수집 JSON 가져오기

모든 시작 방법은 검토 대기 작업을 만든다. 실제 `festivals` 테이블을 즉시 수정하지 않는다.

## 편집 단계

1. 기본정보 관리
2. 라인업 관리
3. 티켓 관리
4. 검토 메모 확인
5. 저장 또는 승인

저장은 작업 초안만 갱신한다. 승인은 DB 트랜잭션을 실행해 정식 축제와 하위 정보를 한 번에 반영한다.

## 축제 JSON 자동 매칭과 병합

축제 JSON을 가져오면 검토 대기 작업을 먼저 찾는다.

- `normalized_name + 시작일 + 종료일`이 정확히 같을 때만 자동으로 같은 작업으로 판단
- 정확한 매칭에 실패하면 같은 시작일·종료일을 가진 검토 대기 후보를 표시하고 관리자가 병합 대상을 직접 선택
- 날짜만 같은 후보는 자동 병합하지 않으며 별도 신규 작업 생성도 직접 선택 가능
- 가져온 JSON의 `normalized_name`은 병합 전에 화면에서 수정 가능
- 승인 후에는 실제 `festivals`까지 같은 `normalized_name + 시작일 + 종료일`을 검사해 중복 신규 작업 생성을 차단
- 정확한 값이 달라도 같은 날짜의 정식 축제를 후보로 표시하며, 별도 신규 작업은 관리자가 명시적으로 선택
- 일치하는 작업이 하나면 기본정보·라인업·티켓 차이를 미리 표시
- 일치하는 작업이 없으면 신규 검토 대기 작업으로 저장
- 같은 조건의 작업이 여러 개면 자동 반영하지 않고 중복 정리를 요청

비교 상태는 다음과 같다.

- `동일`: 공백·줄바꿈·일부 문장부호·가격 표기·URL 추적값 차이를 정리한 뒤 같음
- `표현 차이`: 설명·프로그램 문장의 포함관계 또는 문자 조각 유사도가 높음
- `추가`: 현재 값이 비어 있거나 신규 아티스트·티켓임
- `변경 확인`: 현재 값과 신규 값이 모두 존재하며 의미 있는 차이가 있음

`안전한 변경 모두 반영`은 비어 있는 기본정보와 신규 라인업·티켓만 자동 병합한다. 표현 차이와 실제 변경은 기존 값을 유지하고 검토 메모에 현재값과 신규값을 기록한다. JSON에 없는 기존 라인업은 삭제하지 않는다.

동일 항목과 변경 확인 항목은 반영 전에 상세 내용을 볼 수 있다. 변경 확인 및 표현 차이는 항목마다 `현재 값 유지` 또는 `신규 값 사용`을 선택하며, 선택하지 않으면 현재 값을 유지한다.

## 상태

- 검토 대기: 작성 또는 수정 중인 작업
- 승인: 정식 등록이 완료된 작업
- 거절: 반영하지 않기로 한 작업
- 전체: 모든 작업 이력

## 화면 원칙

- 흰색 배경을 기본으로 사용한다.
- 카드와 편집 영역은 회색 테두리와 약한 그림자로 구분한다.
- 선택 항목은 하늘색 배경 없이 진한 회색 테두리로 표시한다.
- 주요 실행 버튼만 짙은 색을 사용한다.

## 다음 개선

- 기존 페스티벌에 들어온 추가 수집자료를 신규 등록이 아닌 변경안으로 연결
- 2차 라인업을 기존 라인업 유지 + 신규 아티스트 추가 방식으로 반영
- 처리 출처와 변경 이력 조회
